> 依「中立角色 — 不給予判斷,不加入無法源依據的內容」核心原則(CLAUDE.md §0),本檔持續累積審查紀錄。

## 1. SOP — 新增情境卡 / caveats / flow 結論的審查流程

新增任一條結論性陳述前:

1. 列出**真實 primary_ids**(必須在 `nodes.json` 找得到)
2. **同時搜 A/B/C/D 全分類**(D 類雖位階低但是法源,別漏掉)
3. 讀對應節點的 summary + body_plain
4. 確認結論的**每一個限制詞 / 數字 / 條件**都能在條文中對應到原文(可同義轉述,但語意不可超出)
5. 不滿足 (4) 就**不要寫**該結論;若使用者要求保留,改中立描述 + 標「本資料庫未收錄對應函釋」
6. 若僅 D 類有依據而 A/B/C 無,subtitle / caveats 應註明「依 Q○○問答集說明」讓使用者知道位階

可考慮寫一支 `05_scripts/_audit_scenario_sources.py` 自動跑此檢查,定期警告。

## 2. 法源位階(由高至低)

判斷一條結論是否「有法源」時,以下類別都算正式法源(出處皆主計總處或行政院),但**位階不可顛倒**:

1. **A 類核心法規**(《○○出差旅費報支要點》《政府支出憑證處理要點》等)— 最高位階
2. **B 類支出標準**(日支生活費表、雜費限額表、保險表等行政院公告)— 次高
3. **C 類解釋函令**(主計總處解釋彙編)— 補充 A/B 適用疑義,**不可超越 A/B 文義**
4. **D 類問答集**(經費結報常見疑義問答集)— 主計總處出版,**位階最低,不可與 A/B/C 衝突**;若 D 與 A 衝突,以 A 為準

D 類問答集**確實算法源**(因為是主計總處主動發布之解釋資料),所以情境卡 primary_ids 只指 D-XXX 也是合法依據,**但**:
- 不可僅憑 D 類去推翻 A/B 條文文義
- 若主題在 A/B 已有明文,D 類僅作「適用實例補充」用,不應作為「擴大」或「縮減」結論的依據
- 若主題 A/B/C 都沒提,只 D 有,可作為情境卡法源,但 caveats / subtitle 應註明「依 D-XXX 問答集說明」

## 6. 違反原則的處理 SOP

違反此原則的既有內容(無法源支撐的卡 / caveats / flow conclusion)發現後**列入待處理清單**,不自行修改;等人類拍板處理方式(刪 / 改中立 / 補法源)後執行。

正確處理方式有三種(擇一):
- (a) 整張卡 / 段落刪除,等收到法源後再補
- (b) 改寫成**中立描述 + 直接點出「本資料庫尚未收錄相關函釋,實際以主計室審核為準」**,明確告知使用者「無依據可查」
- (c) 從主計總處解釋彙編 / 經費結報問答集找到該函釋,補進 `02_markdown/`,再連回情境卡

## 7. 用語規範

**subtitle / caveats / flow.conclude.note** 用詞須中性化:用「條文規定...」「依○○條...」「本表規定...」「依 Q○○問答集說明...」**取代**「應該...」「不得...」「才能...」「始得...」(除非條文原文就用該語氣)。

AI 助手主動建立或修改情境卡時,**禁止以「常識補強」名義加結論**;若條文沒寫,就不要寫。寧可空白也不要造假結論。
